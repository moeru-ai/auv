## 2. Human Consent: Readiness Refusal Kept As Evidence

MCP call supplied:

```json
{
  "human_gesture_consent": true,
  "human_gesture_timeout_ms": 45000,
  "granted_by": "owner-m0-human-approval"
}
```

Result:

```json
{
  "run_id": "run_1781167666368_87561_0",
  "status": "blocked_not_ready",
  "promotion_artifact_id": "artifact_0003",
  "decision_artifact_id": "artifact_0004",
  "execution_artifact_id": "artifact_0006",
  "promotion_refusals": []
}
```

Inspect evidence:

```text
candidate.action.command.consent.approved human gesture approval granted via local_auth_device_owner_authentication by owner-m0-human-approval
Candidate Action Execution Lineage:
- artifact=artifact_0006 status=blocked_not_ready ... readiness=not_ready blocker=frontmost window 4823 does not match target window 19747
  consent=consent-mcp-m0-1781167666367-execution by=owner-m0-human-approval consent_provenance=human_gesture consent_grade=human_approved
```

This proves the validator was not loosened for the demo. The run reached
human-approved execution consent, then readiness blocked input delivery because
the frontmost window did not match the promoted candidate window.

The follow-up bug fix was not a validator relaxation. It made the
candidate-action activation helper wait until System Events reports the target
bundle as frontmost before allowing the existing readiness check to run.
