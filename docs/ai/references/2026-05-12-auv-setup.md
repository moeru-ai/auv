# AUV Brainstorming Notes

Date: 2026-05-12

Status: draft brainstorming summary

## Product Positioning

AUV stands for Application Use Via. The project is a runtime and description
system for application-specific agent skills.

AUV should not be framed as another generic computer-use agent. The project
does not implement the LLM agent itself. Instead, it provides tool surfaces,
artifact formats, adapter protocols, and execution runtime pieces that existing
agents can use to distill, compile, and run application-specific skills.

The central unit is an application skill bundle. A bundle describes how a
specific application, or a family of similar applications, can be perceived,
grounded, controlled, verified, and recovered.

## Decision Biases And Preferences

The discussion established several decision preferences that should guide the
project.

### Capability-First, Not Platform-First

AUV should avoid designing around a fixed GUI platform, toolkit, or automation
API. The first design question should be: what grounding and control
capabilities are available for this application in this environment?

This means AUV should be able to support traditional desktop applications,
browser applications, Electron applications, mobile devices, game engines, and
custom remote environments without treating any one of them as the default.

### Strongest Available Signal Wins

AUV should not force the lowest common denominator. If a target application has
a reliable extension API, debugger, CDP surface, CLI, or RPC bridge, AUV should
use it. If those are unavailable or insufficient, AUV should fall back to OCR,
CV segmentation, YOLO-based detection, color-based segmentation, and raw input.

The preferred method is the method that is most stable, observable, and
verifiable for the current action.

### Screenshot-First For The First Phase

The first phase should intentionally bias toward screenshot-first grounding.

This is not because screenshots are always best, but because many applications
do not expose useful AX trees, DOMs, APIs, or command surfaces. Starting with
OCR and CV-based segmentation keeps the first architecture honest about hard
applications.

AX tree, CDP, AppleScript, CLI calls, app extensions, and app-specific probes
should be treated as optional stronger signals rather than assumptions.

### AUV Does Not Implement The Agent

AUV should expose MCP and CLI tool surfaces that existing agents can call. The
agent is responsible for reading attempt results, reasoning about tradeoffs,
and composing candidate skill pools during distillation.

AUV is responsible for making attempts executable, observable, reproducible,
and compilable into reusable artifacts.

### Distillation Produces Artifacts

The project should not assume that application skills are fully hand-written.
The preferred workflow is to let an agent, assisted by AUV tools, attempt
multiple grounding and control strategies, inspect the results, and distill the
best strategies into reusable artifacts.

The reusable output should preserve the reasoning-relevant evidence:

- what was attempted
- which signals worked or failed
- which strategy was selected
- which adapters are required or optional
- how success is verified
- how fallback should happen

### Runtime Should Be Pluggable

AUV should not own every OCR engine, YOLO model, OpenCV script, browser bridge,
device bridge, or app-specific probe. External adapters should be able to join
the runtime and declare the capabilities they provide.

This makes AUV a scheduler and artifact runtime rather than a monolithic
automation stack.

### Adapter Composition Over Single Implementation

An action may need multiple adapters at the same time. For example, the action
may use OCR for text, YOLO for icon detection, OpenCV for boundary refinement,
keyboard input for control, and screenshot deltas for verification.

The preferred abstraction is a distilled strategy tree, not a list of unrelated
complete implementations.

### Version Compatibility Must Be Explicit

Artifacts should track compatibility at four layers:

- spec API version
- AUV runtime version
- external adapter versions
- target application version

Spec API mismatch should fail early. Target application mismatch should warn or
trigger revalidation. Adapter mismatch should degrade, fallback, or fail based
on dependency policy.

### Schema Fields Should Use camelCase

Artifact and schema fields should use `camelCase`. CLI command names, package
names, and file names can follow the conventions of their own ecosystem.

## Consensus, Rejections, And Tensions

This section records not only the final direction, but also what was rejected
or corrected during the discussion. These points are important because they
define the design boundaries.

### Agreed Decisions

The discussion reached consensus on these points:

- AUV should not be a generic computer-use framework with only low-level
  mouse, keyboard, and screenshot primitives.
- AUV should focus on application-specific skill bundles.
- AUV should not constrain itself to a specific GUI platform, accessibility
  model, browser engine, mobile platform, or game engine.
- The first phase should bias toward screenshot-first grounding, especially OCR
  and CV-based segmentation.
- AX tree should be optional. It is useful when available, but many
  applications either do not expose it or expose too little useful information.
- AUV should use the most reliable available signal for each application and
  action.
- Runtime adapters should be pluggable, including external OCR, YOLO, OpenCV,
  CDP, CLI, device, extension, or app-specific adapters.
- Distillation and execution should be separate workflows.
- AUV should expose MCP and CLI tool surfaces for existing agents instead of
  implementing the agent itself.
- Distillation should produce reusable artifacts that capture attempts,
  selected strategies, dependencies, verification rules, and version
  constraints.
- Schema fields should use `camelCase`.
- `apiVersion` should be top-level.
- Dependencies should be listed separately from version constraints.

### User Rejected Or Corrected

The following earlier directions were rejected or corrected:

- A runtime-only framing was inaccurate. AUV needs at least two distinct
  phases: distill/compile and run. The distill/compile phase uses an
  external agent that calls AUV tools, receives attempt results, and composes
  reusable skill artifacts.
- AUV should not be described as if it makes all high-level agent decisions.
  Existing agents should do the reasoning and orchestration during
  distillation.
- AUV should not assume that one action is simply a list of manually authored
  complete implementations. It may be better to view an action as a distilled
  strategy tree that combines multiple adapters and fallback signals.
- Version compatibility should not be hidden inside a nested `compatibility`
  block. The preferred shape is simpler: top-level `apiVersion`, a `versions`
  block, and a separate `dependencies` block.
- Schema examples should not use snake_case. They should use `camelCase`.
- AUV should not depend only on built-in capabilities. External adapters should
  be able to dynamically provide models, scripts, probes, and detection
  capabilities.

### Assistant Reservations Or Design Pushback

The following are design cautions raised during the discussion:

- Pure autonomous distillation may be too expensive or unreliable as the only
  workflow. A practical path may combine record, annotate, agent-assisted
  distillation, replay, and evaluation.
- Runtime fallback should still exist even if distillation selected the best
  known strategy, because application state, versions, permissions, and UI
  layouts can change.
- AUV should avoid becoming a monolithic automation stack that owns every OCR
  engine, detector, model, bridge, and control surface.
- The first phase should avoid over-investing in a universal UI detector before
  proving music-player workflows with OCR, CV segmentation, and targeted
  app-specific refinements.
- The public model should stay application-centered, even though the internals
  may support environment adapters. Users think in terms of "use Apple Music"
  or "use VSCode", not first in terms of platforms or channels.

### Still Unsettled

These points are not fully decided:

- Whether the public bundle term should be `SkillBundle`,
  `ApplicationBundle`, or another name.
- Whether the distillation output is one file, a directory, or a package with
  traces and media assets.
- How much fallback should be compiled into artifacts versus rediscovered at
  runtime.
- Whether the first adapter protocol should prioritize short-lived CLI calls,
  long-running sidecars, or both from day one.
- How strict target application version mismatch should be during execution.
- Which three platforms should be selected for the first music-player push.

## Platform Principle

AUV is GUI-platform agnostic.

It does not assume that an application exposes a DOM, AX tree, MCP server, CLI,
or stable native automation API. Any environment can be supported if it can
provide one or more grounding channels and one or more control channels.

Grounding channels may include:

- screenshots
- OCR
- CV segmentation
- YOLO or other object detection models
- accessibility trees
- browser DOM or CDP
- game engine debuggers or inspectors
- app-specific state probes
- remote device streams
- VNC or RPC bridges

Control channels may include:

- mouse and keyboard input
- touch input
- hotkeys
- AppleScript
- CLI calls
- CDP
- browser extensions
- VSCode extensions
- ADB
- XCTest
- system hooks
- game engine debugger commands
- RPC
- app-specific automation bridges

AUV chooses the most reliable available control surface for each application.
It does not prescribe one universal grounding method.

## Initial Application Direction

The first application family should be music players.

Initial target applications include:

- Apple Music
- NetEase Cloud Music
- other similar music players where the same domain actions can be compared
  across multiple platforms

This family is a useful first target because the domain actions are stable, but
the available automation surfaces differ heavily between applications and
platforms.

Example domain actions:

- search for a track
- play a track
- pause playback
- skip to next track
- add a track to a playlist
- like or favorite a track
- read the current playing track

## Screenshot-First Phase

The first implementation phase should prefer screenshot-first grounding.

The initial grounding stack should prioritize OCR and CV-based region
segmentation, with app-specific refinements where needed. AX tree, CDP,
AppleScript, CLI calls, or app-specific APIs are useful enhancements, but they
must not be assumed to exist.

The first phase should focus on:

- screenshot capture
- OCR text extraction
- CV-based region and boundary detection
- color-based segmentation where OCR is insufficient
- YOLO or icon detection where visual controls are icon-only
- coordinate alignment and clickable-area refinement
- verification through screenshot deltas, OCR predicates, visual state changes,
  or external state probes

The guiding principle is that AUV should use the strongest available signal,
not the most generic signal.

## Distill, Compile, And Run

AUV should distinguish between three major workflows.

### Distill Workflow

The distill workflow is used to discover working skills.

An existing LLM agent calls AUV through MCP or CLI tools. The agent uses these
tools to inspect applications, attempt actions, compare grounding strategies,
read attempt results, and decide which strategies are worth keeping.

AUV should expose tools such as:

- `auv inspect`
- `auv screenshot`
- `auv ocr`
- `auv segment`
- `auv detect`
- `auv attempt`
- `auv replay`
- `auv verify`
- `auv compile`

AUV does not decide everything by itself. The agent receives structured results
from AUV, reasons about them, and composes the final skill pool.

### Compile Workflow

The compile workflow turns distilled skill pools into reusable artifacts.

Inputs may include:

- source observations
- screenshots
- OCR outputs
- segmentation outputs
- object detection outputs
- attempt traces
- selected grounding strategies
- selected control strategies
- verification predicates
- fallback policies
- dependency requirements
- target application version constraints

The output is a reusable skill artifact that can be loaded later by the AUV
runtime.

### Run Workflow

The run workflow executes compiled artifacts.

The AUV runtime loads a skill bundle, checks compatibility, discovers available
adapters, schedules required probes and actions, executes distilled recipes,
verifies results, and emits traces for future re-distillation.

## Runtime Adapter Model

Runtime adapters should be pluggable.

External adapters should be able to provide additional perception, grounding,
control, verification, or recovery capabilities. AUV itself should not need to
own every OCR engine, YOLO model, OpenCV script, browser bridge, or device
bridge.

Examples of adapters:

- OCR adapter
- YOLO adapter
- OpenCV region detector
- AX tree adapter
- CDP adapter
- browser extension adapter
- CLI adapter
- ADB adapter
- iOS remote device adapter
- VSCode extension adapter
- Godot debugger adapter
- Unity inspector or mod adapter

Adapters should declare:

- id
- version
- provided capabilities
- accepted inputs
- output schema
- permissions
- runtime requirements
- latency or cost hints
- cacheability
- app or platform constraints

The runtime can then decide which adapters to call based on the current target
application, available dependencies, and distilled recipe requirements.

## Adapter Composition

An action may use multiple adapters together.

For example, a `playTrack` action might use:

1. screenshot capture to observe the current UI
2. OCR to identify search results
3. keyboard input to focus search
4. YOLO or icon detection to find the play button
5. OpenCV alignment to refine the clickable area
6. screenshot delta or progress-bar motion to verify playback

This is not necessarily multiple complete implementations of the same action.
It is better modeled as a distilled strategy tree composed from multiple
available capabilities.

## Distilled Recipe Model

The output of distillation should be a distilled recipe.

A distilled recipe describes:

- grounding strategy
- control strategy
- verification strategy
- fallback policy
- required and optional adapters
- expected inputs and outputs
- target application constraints
- known failure modes

Terminology:

- Adapter: a channel that provides a capability.
- Probe: an operation that observes state or extracts information.
- GroundingStrategy: how to locate or identify a target in the current scene.
- ControlStrategy: how to operate the target.
- VerificationStrategy: how to determine whether an action succeeded.
- DistilledRecipe: the reusable action strategy produced by distillation.
- FallbackPolicy: how to degrade or retry when the preferred strategy fails.

## Chromium And Electron Applications

Chromium-compatible and Electron applications should support multiple possible
surfaces.

Possible support order:

1. native app protocol or extension API
2. CDP attach or remote debugging protocol
3. browser extension bridge
4. DOM and accessibility snapshot
5. screenshot, OCR, and CV grounding
6. raw mouse and keyboard fallback

The best path depends on the target application.

For VSCode, for example, the most reliable path may be a VSCode extension,
followed by command palette and keybindings, then Electron/CDP inspection,
then screenshot-first fallback.

Browser extensions can be implemented with tools such as WXT. They should be
treated as in-process grounding and control bridges, not as a browser-only
feature.

## Mobile And Device Support

Android and iOS should be treated as environments with their own possible
grounding and control channels.

Android support may use:

- screenshots
- OCR and CV
- ADB
- accessibility services
- system hooks
- app-specific bridges

iOS support may use:

- remote device screenshot streams
- VNC-like control
- XCTest
- RPC bridges
- usbmux tunnels
- system hooks where available

The `vphone-cli` project is a useful reference for iOS remote-device style
control and inspection:

https://github.com/Lakr233/vphone-cli

## Versioning Model

AUV should track compatibility across four layers.

1. Spec API version: the version of the artifact description language and
   operation definitions. If this is incompatible, the artifact may not be
   loadable at all.
2. AUV runtime version: the version of the runtime required to execute the
   artifact.
3. Adapter versions: external adapter versions required by the artifact.
   Internal adapters can follow the AUV runtime version.
4. Target application version: the version range of the application that the
   artifact was distilled or validated against.

If the target application version does not match, AUV should warn and may
require revalidation. If the spec API version does not match, AUV should fail
early.

Schema fields should use camelCase.

Example shape:

```yaml
apiVersion: auv.ai/v1alpha1
kind: SkillBundle

metadata:
  id: music.appleMusic
  name: Apple Music
  version: 0.1.0

target:
  application:
    id: com.apple.Music
    name: Apple Music

versions:
  auv: ">=0.1.0 <0.2.0"
  targetApplication: ">=1.5.0 <1.6.0"

dependencies:
  adapters:
    - id: musicYolo
      version: "^0.3.0"
      required: false
      onMissing: degrade
      provides:
        - ui.objectDetection
        - music.controlIconDetection

    - id: opencvRegionDetector
      version: ">=0.2.0"
      required: false
      onMissing: fallback
      provides:
        - ui.regionSegmentation
        - ui.boundaryDetection

  tools:
    - id: osascript
      required: false
      onMissing: fallback

capabilities:
  grounding:
    - screenshot
    - ocr
    - cvRegions
    - iconDetection
  control:
    - keyboard
    - mouse
    - cli
  verification:
    - screenshotDelta
    - ocrPredicate
    - stateProbe
```

## Review Update: Runtime, Drivers, Runs, And Observability

The later discussion refined the boundary between AUV core, built-in drivers,
runtime calls, CLI commands, run recording, replay, and inspection. These notes
should be treated as design input for the next spec pass, not as an
implementation plan.

Important naming note: names in this section are provisional. Terms such as
`driver`, `RunStore`, `ArtifactStore`, `run`, `stop`, `resumeToken`,
`trace replay`, `live replay`, and `inspect UI` are working names used to make
the discussion concrete. They are not final API or product names. The team
should revisit and approve naming before implementation, because these names
will affect public APIs, CLI commands, docs, artifact schemas, and long-term
developer ergonomics. Avoid treating these names as settled too early, or the
project may pay unnecessary migration and rework cost.

### Product Boundary

AUV should not be a thin core orchestrator that only coordinates external
plugins. It should provide a useful application-command runtime with built-in
first-party drivers.

The preferred framing is:

> AUV is an application command runtime that turns application UI workflows into
> command-like, inspectable, replayable, and eventually shortcut-like
> operations.

The public experience should make UI operations feel more like commands:

- `auv-apple-music playlist ls`
- `auv-apple-music play my-favorite --playlist`

However, CLI commands are only one frontend. The central execution surface
should be a library/runtime API that can also be used by MCP tools, shortcuts,
tests, applications, and app-specific command wrappers.

### Responsibilities

AUV core is responsible for:

- artifact and recipe schema
- command invocation and runtime execution
- implicit run creation for every invocation
- recipe step scheduling
- driver capability resolution
- run event recording
- artifact capture and retrieval
- stop and resume boundaries
- trace replay and live replay
- inspect APIs and inspect UI support
- compatibility checks
- resource management for retained run data

AUV first-party drivers are responsible for providing built-in grounding,
control, verification, and observation capabilities. These drivers should be
officially supported AUV functionality, not merely examples.

Expected first-party driver areas include:

- screenshot capture
- mouse, keyboard, and touch control
- OCR
- CV segmentation
- icon or YOLO-style detection
- CDP
- browser extension control
- ADB
- platform automation such as AppleScript or AX where appropriate

Custom or third-party drivers should use the same driver protocol and should be
able to extend, override, or replace first-party capabilities.

### Driver Boundary

The project should likely use the term `driver` rather than `adapter` for the
runtime capability provider concept, but this naming is not final.

A driver may contain multiple internal implementation backends. For example, an
OCR driver may choose between multiple OCR models, local processes, remote
services, app-specific fetchers, or UI-specific adapters. Those backend details
should not become the top-level AUV command model.

The runtime and traces still need to record enough detail to explain what
happened:

- which driver was called
- which driver operation was called
- which inputs were used
- which backend or model actually produced the result when relevant
- which outputs or artifacts were produced
- how those outputs were used by the recipe
- timing, confidence, and error information

This preserves debuggability without forcing every command or artifact to bind
itself to one specific backend implementation.

### Commands, Runtime API, And Library Calls

CLI must not become the architecture center.

Every app-specific CLI command should be a frontend over the same runtime API
that library users and MCP tools can call directly. Conceptually:

```ts
const result = await auv.invoke({
  command: "appleMusic.playlist.list",
  input: {},
  target: { applicationId: "com.apple.Music" }
});
```

The runtime API should support at least:

- `invoke`
- `resume`
- `inspect`
- `replay`

CLI, MCP, app-specific wrappers, tests, shortcuts, and embedded library usage
should all go through this shared runtime boundary.

### Implicit Runs

Every invocation should implicitly create a run. The caller should not need to
create a run first, but every result should be inspectable through a `runId` or
equivalent reference.

This applies equally to:

- CLI calls
- library calls
- MCP tool calls
- shortcut-like invocations
- replay and test invocations

Runs are not a CLI convenience. They are part of the execution contract because
AUV needs to make UI automation observable, inspectable, replayable, and
re-distillable.

### Run Store And Artifact Store

A phase-one implementation should include a run storage abstraction. The exact
name is still open, but `RunStore` is the current preferred term.

`RunStore` and `ArtifactStore` are provisional names. The team should decide
whether these names describe the right product and API concepts before writing
stable interfaces around them.

The store needs to retain:

- run metadata: command, input, target, timestamps, runtime version
- step tree: recipe steps and nested driver operations
- event log: observations, actions, verification attempts, fallbacks, stops
- outputs: command-level returned data
- stop state: resumable boundaries and required input shape
- artifact references: screenshots, crops, OCR outputs, masks, snapshots
- diagnostics: backend identity, timing, confidence, errors, fallback path

Internally, this may be split between a structured event store and an artifact
store. Externally, it can still appear as one run inspection surface.

Likely implementations include:

- in-memory store for tests
- file-backed local store for CLI and early development
- SQLite-backed local store for longer-term local usage
- remote store for daemon, MCP server, or team debugging use cases

### Observability And Replay

Observability and replay should be phase-one infrastructure, not later
diagnostic features.

The guiding principle is:

> Every invoke produces a replayable run.

A driver operation should not be considered complete until its important
inputs, outputs, artifacts, and replay policy have been recorded.

AUV should distinguish at least two replay modes:

1. Trace replay: do not touch the real UI. Reuse recorded observations and
   artifacts to replay recipe logic, grounding, parsing, and verification.
2. Live replay: re-run the recipe against the real target application and
   collect a fresh trace.

Trace replay is important for debugging, tests, regression checks, and
distillation refinement. Live replay is important for validating that a recipe
still works against the current application state and version.

`Trace replay` and `live replay` are provisional labels. The underlying
distinction is important, but the final names should be reviewed before they
become CLI flags, API enum values, or documentation terms.

### Inspect UI

The project should include an inspect UI early. Rich traces are much less useful
if developers can only inspect them as raw JSON.

`Inspect UI` is also a working name. The concept may eventually become a local
viewer, devtools surface, trace explorer, or another productized interface.

The inspect UI should eventually support:

- run list
- command, input, target, and version metadata
- step tree or timeline
- driver operation details
- before and after screenshots
- OCR text overlays
- detection boxes
- segmentation masks
- clicked coordinates and target rectangles
- verification predicates and results
- stop or resume boundary details
- trace replay and live replay controls
- artifact export or run bundle export

The CLI should still support text and JSON inspection, but the UI should be a
first-class development surface.

Possible commands:

```bash
auv inspect run_123
auv inspect run_123 --json
auv inspect run_123 --ui
auv replay run_123 --mode trace
auv replay run_123 --mode live
```

The canonical inspection entry point should probably be `auv inspect`.
App-specific CLIs may offer forwarding conveniences, but should not own the
inspection model.

### Stop And Resume Boundaries

The runtime result protocol should not leak the concept of an agent. It should
describe the execution boundary neutrally.

Instead of returning a state such as `needsAgent`, AUV should describe the kind
of stop encountered:

- selection required
- additional input required
- permission or login required
- confirmation required
- unsupported capability
- ambiguous target
- blocked external condition
- unrecoverable failure

The exact result schema is still unsettled. One possible direction is:

- `state`: high-level execution state such as completed, stopped, or failed
- `outputs`: command-level returned data
- `runId`: inspection reference
- `stop`: structured stop boundary, present only when stopped

The `stop` object may include a resume token and required input shape if the
run can continue. The term `continuation` may be too abstract; `resume` or a
resume token inside `stop` may be clearer.

`stop`, `resume`, and `resumeToken` are provisional protocol names. They should
not be treated as settled until the result model is reviewed against CLI,
library, MCP, shortcut, and UI use cases.

This model should be friendly to agents, users, scripts, and UI clients without
coupling the runtime protocol to any one caller type.

### Runtime Return Format

YAML should not be assumed as the runtime response format. The current split is:

- artifact/spec authoring: YAML or JSON may both be useful
- CLI default output: human-readable text or tables
- CLI machine output: JSON through a flag such as `--json`
- MCP result: structured JSON object
- library result: typed objects
- long-running event streams: possibly JSONL or a typed event stream
- stored traces: structured events plus artifact references

The runtime result schema is still an open design area and should be specified
before implementation.

### Resource Management

AUV should preserve rich run evidence by default, then manage cost through
policy rather than dropping important data too early.

Useful data to retain includes:

- full screenshots
- cropped regions
- OCR raw output
- OCR normalized output
- detection boxes
- segmentation masks
- clicked coordinates
- target rectangles
- before and after screenshots
- verification snapshots
- driver inputs and outputs
- backend or model identity
- logs, errors, timing, and confidence
- recipe step tree
- stop and resume boundaries
- final outputs

Resource management should include:

- retention policy
- artifact classification such as output, evidence, debug, or temporary
- image compression
- text compression
- deduplication by content hash
- redaction for sensitive text or screenshot regions
- sampling for long-running flows
- pruning policies for successful runs
- fuller retention for failed or pinned runs
- export and import of debug bundles
- pinning important runs so they are not cleaned up

OpenTelemetry may be useful as an export or integration target, but should not
be the primary AUV trace data model at first. AUV traces include UI artifacts,
screenshots, OCR, detection boxes, and recipe-specific decisions that need a
domain-specific model. These can later be mapped to OTel spans and events.

### Non-Goals And Boundary Guards

AUV should not:

- implement the LLM agent itself
- make CLI the only supported invocation mechanism
- treat run recording as optional debug logging
- require users to manually create runs before invoking commands
- hide all driver internals when they are needed for debugging
- expose every driver backend detail as top-level artifact schema
- collapse all stopped execution states into generic failure
- require one universal UI model across all applications
- depend exclusively on third-party drivers for first useful behavior

### Impact Range

These updates affect several future specs and implementation areas:

- runtime API design
- CLI design
- MCP tool design
- driver protocol
- artifact and recipe schema
- result schema
- run storage
- artifact storage
- trace event model
- replay engine
- inspect UI
- retention and redaction policy
- first-party driver packaging
- test and evaluation strategy

Because these are cross-cutting decisions, the team should review them before
implementation begins.

The discussion added these open questions:

1. Should the runtime result state use `completed`/`stopped`/`failed`, or a

2. What is the exact shape of a structured `stop` object?
3. Should resumability be represented as a `resumeToken`, a checkpoint id, or
   another concept?
4. What should the first RunStore implementation be: file-backed, SQLite, or
   both?
5. How much data should be captured by default for successful runs?
6. Which artifact classes should exist in the retention policy?
7. What should be redacted by default, and how does a user override it?
8. Should the inspect UI be built into the main package or launched as a

9. What is the minimum useful trace replay contract for phase one?
10. How should live replay avoid accidental destructive actions?
11. Which first-party drivers are required for the first useful AUV release?
12. Should OpenTelemetry export be included in phase one or deferred?
13. Which provisional terms in this section should become final public names,

## Review Update: Implementation Flow And Module Priority

This section captures a rough implementation flow for team review. It is not an
approved implementation plan. The goal is to help contributors discuss
responsibilities, sequencing, shared foundations, and proof-of-concept scope
without losing the current reasoning.

All names in this section are provisional. Module names, package names, command
names, API names, and phase labels should be reviewed before implementation.

### Team Shape And Reference Tracks

The expected early development group is about three people. The work should
not be divided as "one person, one unrelated demo app." Instead, the early
tracks should share the same runtime and driver foundations while exercising
different surfaces.

The current preferred tracks are:

1. NetEase Cloud Music application family
2. A separate macOS application case
3. Chrome or browser-surface case, likely on Linux

### NetEase Cloud Music As Primary Grounding Reference

NetEase Cloud Music should be treated as the primary grounding reference, not
just one target application.

The same application family can be examined across:

- macOS
- iOS
- Android
- Web
- Windows
- Linux

The purpose is not to support every platform in the first pass. The purpose is
to study how the same domain actions vary across OS, UI implementation,
automation surface, and available signals.

This track should help answer:

- which parts of a command are domain-level
- which parts are app-version or platform-specific
- how far OCR, CV, and icon grounding can go
- how recipes should vary across target implementations
- how driver selection should work when stronger signals exist
- how to avoid overfitting to AX tree, AppleScript, CDP, or any one platform

NetEase Cloud Music is especially useful because it cannot be assumed to expose
stable AX, AppleScript, CLI, or app-specific automation surfaces everywhere.
That makes it a better pressure test for screenshot-first grounding than an app
that can be solved mostly through a privileged automation API.

### macOS Contrast Track

Another developer may focus on a separate macOS application case.

This track should validate macOS-specific platform capabilities and also test
that AUV does not overfit to NetEase Cloud Music.

Areas to validate include:

- screenshot capture
- window discovery or targeting
- mouse and keyboard input
- macOS permissions
- AppleScript as a first-party driver where supported
- AX or app-specific automation where useful
- interaction between strong signals and screenshot-first fallback

AppleScript should be treated as a special driver category. It can be highly
valuable for supported applications, but it should not become the only model
for macOS automation. Apple Music may be a useful reference, but it may be too
easy to solve through AppleScript or Shortcuts alone, so it should not be the
only macOS proof point.

### Chrome Or Browser Surface Contrast Track

The Linux or browser contrast track may use Chrome as the initial case.

This track should validate browser-surface support rather than replace
screenshot-first work. It should help define how CDP, browser extension
drivers, DOM snapshots, screenshots, OCR fallback, and browser-side evaluation
fit into the same runtime model.

`agent-browser`-style tooling can be used as a reference for capabilities such
as:

- connecting to a browser
- page snapshots
- screenshots
- click and type operations
- console and network inspection
- browser-side evaluation
- CDP-oriented control

However, AUV should keep its own runtime, run recording, replay, inspection,
and driver abstractions. The browser track should not become a direct wrapper
around an existing browser automation tool.

### Shared Principle For Tracks

The early tracks may differ in target application, platform, and available
drivers, but they must share the same execution substrate.

The guiding rule is:

> App tracks may differ in drivers and recipes, but must share command
> invocation, implicit run creation, driver operation recording, artifact
> capture, inspection, and replay semantics from day one.

Without this shared substrate, each track risks becoming an incompatible
automation script demo that is hard to merge back into AUV.

### Priority 0: Shared Execution Skeleton

The first implementation priority should be the shared execution skeleton.

This is required before app tracks diverge too far.

Expected capabilities:

- runtime `invoke` shape
- implicit run creation for every invocation
- driver operation wrapper
- provisional RunStore abstraction
- provisional ArtifactStore abstraction or equivalent artifact layer
- run event append
- artifact reference model
- CLI calling the runtime instead of owning execution logic
- minimal `inspect` command
- minimal fixture or example command

The goal is not complete automation. The goal is to make every contributor use
the same execution, recording, and inspection path from the beginning.

### Priority 1: Minimal Observable UI Automation

The second priority should be the smallest useful UI automation layer that is
observable through runs.

Expected capabilities:

- screenshot capture
- image artifact storage
- OCR driver with one working backend
- mouse and keyboard input driver
- before and after screenshot capture around actions
- operation timing, errors, and confidence where available
- enough inspect output to see each step and artifact reference

This priority is especially important for the NetEase Cloud Music track, but it
should also support the macOS and browser contrast tracks.

### Priority 2: Proof-Of-Concept Commands

The first app commands should be few and deliberately chosen. They should test
the runtime boundaries rather than maximize feature count.

Candidate command set:

- read current playing track or playback status
- search
- play a selected result
- list playlists
- play from a playlist
- one ambiguous or failure-prone command that exercises stop and resume

NetEase Cloud Music should lead this command exploration. The macOS and browser
tracks should implement comparable or analogous actions where possible so the
team can compare whether the abstraction generalizes.

### Priority 3: Trace Replay

Trace replay should be introduced early, not after all app PoCs are complete.

The minimum useful version should:

- load a previous run
- reuse recorded screenshots, OCR results, detection results, or other
  observations
- replay recipe logic without touching the real UI
- reproduce failed runs enough to debug grounding and verification logic
- allow runs to become test fixtures

Live replay can come later, but the interface and event model should leave room
for it.

### Priority 4: Inspect UI

CLI inspection should exist first, but a visual inspect UI should follow early.
It is important for team productivity because image-heavy automation is painful
to debug through raw JSON alone.

The first UI does not need to be polished. It should make runs understandable:

- run list
- command, input, target, and timestamps
- step tree or timeline
- screenshots
- OCR text overlay
- click target or target rectangle
- driver operation details
- stop boundary details
- replay entry points when available

### Priority 5: Strong-Signal First-Party Drivers

After the shared skeleton and observable UI automation path exist, the team can
add stronger first-party drivers according to track needs.

Likely drivers include:

- CDP or browser driver for the Chrome/browser track
- browser extension driver for Web targets
- AppleScript driver for supported macOS applications
- AX driver where useful and available
- ADB driver for Android NetEase Cloud Music
- iOS screenshot/control or remote-device driver for iOS NetEase Cloud Music
- platform-specific window targeting and input refinements

These drivers should be official AUV capabilities, but they must use the same
driver operation recording and run inspection model as the screenshot-first
drivers.

### Suggested Module Priority

The early codebase may eventually need modules in roughly this order:

1. runtime invocation
2. run recording
3. artifact storage
4. driver operation protocol
5. screenshot driver
6. OCR driver
7. input driver
8. inspect CLI
9. simple app command wrapper
10. trace replay
11. inspect UI
12. CDP/browser driver
13. AppleScript or macOS automation driver
14. ADB/mobile drivers
15. richer recipe compilation and distillation workflows

This list is a priority guide, not a package structure. Names and boundaries
should be revisited before implementation.

### PoC Success Criteria

The first useful PoC should demonstrate more than "a command can click a UI."
It should show that AUV's execution model is working.

A successful early PoC should demonstrate:

- at least one NetEase Cloud Music command executed through the runtime API
- the same command available through CLI as a frontend
- an implicit run created for the command
- screenshots and OCR artifacts retained
- driver operations visible in inspect output
- command output returned separately from debug artifacts
- at least one failed or ambiguous run that can be inspected
- trace replay of recorded observations for at least one command path
- enough shared code that another app track can reuse the same runtime path

### Risks

Important risks to watch:

- app tracks diverge into unrelated scripts
- CLI becomes the real architecture instead of a frontend
- run recording is delayed and debugging becomes expensive
- images and OCR artifacts are discarded too early
- NetEase-specific assumptions leak into the core runtime
- macOS automation assumptions leak into cross-platform drivers
- browser/CDP support becomes a separate architecture
- naming is stabilized before the concepts are well understood

### Additional Implementation-Flow Questions

The team should still decide:

1. Which NetEase Cloud Music platform should be attempted first?
2. What is the first non-NetEase macOS application case?
3. Should Chrome on Linux be the first browser contrast track, or should Web
   NetEase Cloud Music cover that surface first?
4. What is the minimum acceptable file-backed run storage format?
5. Which OCR backend should be used for the first working driver?
6. How much inspect CLI is needed before app PoCs begin?
7. How soon should the inspect UI start?
8. Which commands are mandatory for the first PoC review?
9. What data must be captured for every driver operation?
10. Which names are too provisional to encode in stable public APIs?
